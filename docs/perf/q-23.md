# Q23

```sql
SELECT SearchPhrase, MIN(URL), MIN(Title), COUNT(*) AS c, COUNT(DISTINCT UserID) FROM hits WHERE Title LIKE '%Google%' AND URL NOT LIKE '%.google.%' AND SearchPhrase <> '' GROUP BY SearchPhrase ORDER BY c DESC LIMIT 10;
```

## Fast path

`group_fused_q23.rs` — **ClickHouse-style incremental aggregation** (not materialize all `(phrase, user)` pairs):

1. **Pass 1:** 256-way sharded `AHashMap<phrase_hash, count>` — O(distinct phrases), not O(matching rows).
2. Heap top-10 phrase hashes by count.
3. **Pass 2:** one rescan per top phrase — single `AHashSet` at a time for `COUNT(DISTINCT UserID)` + row indices for MIN.

Warm `serve` **evicts** columns not referenced by the current query (`Table::retain_columns`) so Q23 is not stacked on ~33 GiB from Q1–22.

ClickHouse insight: [`Aggregator`](https://github.com/ClickHouse/ClickHouse/blob/master/src/Interpreters/Aggregator.cpp) updates compact **per-group aggregate states** in a hash table; for `ORDER BY count LIMIT K`, a two-phase count-then-detail pass avoids storing every row pair.
